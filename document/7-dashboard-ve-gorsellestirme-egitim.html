📊 Grafana: REST API Performance Dashboard

Dashboard Oluşturma Adımları

  1. Grafana'ya giriş yapın (admin/admin)
  2. Dashboards → New → New Dashboard
  3. Aşağıdaki panelleri sırasıyla ekleyin:

Panel 1: HTTP İstek Oranı (Time Series)

Title: "HTTP İstek Oranı (req/s)"
Type: Time Series
Query A: rate(http_server_request_duration_seconds_count[1m])
Legend: {{http_request_method}} {{url_scheme}}://{{server_address}}{{http_route}}

Panel 2: Response Süresi Percentile (Time Series)

Title: "Response Süresi (saniye)"
Type: Time Series

Query A (p50):
  histogram_quantile(0.50, rate(http_server_request_duration_seconds_bucket[5m]))
  Legend: p50

Query B (p95):
  histogram_quantile(0.95, rate(http_server_request_duration_seconds_bucket[5m]))
  Legend: p95

Query C (p99):
  histogram_quantile(0.99, rate(http_server_request_duration_seconds_bucket[5m]))
  Legend: p99

Panel 3: Toplam İstek Sayısı (Stat)

Title: "Toplam İstek Sayısı"
Type: Stat
Query: sum(http_server_request_duration_seconds_count)
Color mode: Background

Panel 4: HTTP Durum Kodları (Pie Chart)

Title: "HTTP Status Code Dağılımı"
Type: Pie Chart
Query: sum by(http_response_status_code)(
  increase(http_server_request_duration_seconds_count[5m])
)
Legend: {{http_response_status_code}}

Panel 5: .NET Runtime Metrikleri (Gauge + Time Series)

Title: "GC Heap Bellek (MB)"
Type: Gauge
Query: process_runtime_dotnet_gc_heap_size_bytes / 1024 / 1024
Unit: MB
Min: 0, Max: 500

Title: "GC Collections"
Type: Time Series
Query: increase(process_runtime_dotnet_gc_collections_count[1m])
Legend: Gen {{generation}}

Panel 6: Aktif HTTP Bağlantıları (Time Series)

Title: "Aktif HTTP Bağlantıları"
Type: Time Series
Query: http_server_active_requests

📈 Kibana: Log Analiz Dashboard

Data View Oluşturma

  1. Stack Management → Data Views → Create data view
  2. Name: RestApi Logs
  3. Index pattern: restapi-logs-*
  4. Timestamp field: @timestamp

Kibana Dashboard Panelleri

Panel 1: Log Hacmi (Bar Chart)

  1. Dashboard → Create visualization
  2. Visualization type: Bar vertical
  3. Horizontal axis: @timestamp (Date histogram)
  4. Vertical axis: Count
  5. Break down by: level field

Panel 2: Hata Dağılımı (Donut Chart)

  1. Visualization type: Donut
  2. Slice by: level field
  3. Metric: Count

Panel 3: En Çok Hata Veren Endpoint'ler (Data Table)

  1. Visualization type: Table
  2. Filter: level: "Error" OR level: "Warning"
  3. Rows: Top values of fields.RequestPath
  4. Metric: Count

Panel 4: Son Hatalar (Saved Search)

  1. Discover seçin
  2. Filter: level: "Error"
  3. Columns: @timestamp, level, message, fields.RequestPath, fields.StatusCode
  4. Save as panel to dashboard

🎭 Canlı Demo Scripti

Hazırlık

# Terminal 1: Docker servisleri
cd RestApi
docker compose up -d
docker compose ps

# Terminal 2: .NET uygulaması
cd RestApi/RestApi
dotnet run

# Tarayıcı sekmeleri aç:
# 1. Grafana  → http://localhost:3000
# 2. Kibana   → http://localhost:5601
# 3. Jaeger   → http://localhost:16686
# 4. Terminal → Trafik üretici

Senaryo 1: Normal Trafik

# Birkaç normal istek gönder
for i in $(seq 1 20); do
  curl -s http://localhost:5000/api/products > /dev/null
  sleep 0.5
done

echo "✅ Normal trafik gönderildi"
# → Grafana'da düşük ve stabil istek oranı görülür
# → Kibana'da Information seviyesinde loglar görülür
# → Jaeger'da başarılı trace'ler görülür

Senaryo 2: Yük Testi

# Yüksek trafik oluştur
for i in $(seq 1 200); do
  curl -s http://localhost:5000/api/products > /dev/null &
done
wait

echo "✅ Yük testi tamamlandı"
# → Grafana'da istek oranı spike görülür
# → Response süresi artar (p95, p99)
# → GC Heap kullanımı artar
# → Aktif bağlantı sayısı yükselir

Senaryo 3: Hata Üretimi

# 404 hataları üret
for i in $(seq 1 50); do
  curl -s http://localhost:5000/api/products/$((RANDOM % 10000 + 1000)) > /dev/null
done

# 401 hataları üret (yetkisiz erişim)
for i in $(seq 1 30); do
  curl -s http://localhost:5000/api/notes > /dev/null
done

# Geçersiz JSON ile 400 hatası
for i in $(seq 1 20); do
  curl -s -X POST http://localhost:5000/api/products \
    -H "Content-Type: application/json" \
    -d '{"invalid": true}' > /dev/null
done

echo "✅ Hata senaryoları tamamlandı"
# → Grafana'da HTTP status code dağılımı değişir
# → Kibana'da Error ve Warning logları artar
# → Jaeger'da hatalı trace'ler kırmızı görünür

Senaryo 4: Sorun Analizi (Root Cause Analysis)

# 1. Grafana'da HTTP 500 hatalarının arttığı zamanı belirle
#    → Status code pie chart'ta 500'lerin oranına bak

# 2. Kibana'da o zaman aralığındaki ERROR loglarını filtrele
#    → level: "Error" AND @timestamp > "..."
#    → Hata mesajından detay al

# 3. Hata logundaki TraceId'yi kopyala

# 4. Jaeger'da TraceId ile arama yap
#    → Hangi span'de hata oluştuğunu gör
#    → Span detaylarından root cause'u belirle

# Bu akış "single pane of glass" monitoring yaklaşımını gösterir:
# Metrik (Grafana) → Log (Kibana) → Trace (Jaeger)

🔔 Grafana Alerting (Uyarı Sistemi)

Alert Rule Oluşturma

  1. Alerting → Alert rules → New alert rule
  2. Rule name: High Error Rate
  3. Query: 
    # Hata oranı (%5'ten fazlaysa uyarı)
sum(rate(http_server_request_duration_seconds_count{http_response_status_code=~"5.."}[5m]))
/
sum(rate(http_server_request_duration_seconds_count[5m]))
> 0.05
  4. Evaluation: Every 1m for 5m
  5. Condition: when last() of A is above 0.05

Örnek Alert Kuralları

Alert Koşul Önem
High Error Rate HTTP 5xx oranı > %5 🔴 Critical
Slow Response p95 response time > 2 saniye 🟡 Warning
High Memory GC Heap > 300MB 🟡 Warning
Service Down Prometheus target UP = 0 🔴 Critical

⭐ Production Best Practices

Monitoring

  • Her servis için RED metrikleri izleyin: Rate, Error rate, Duration
  • SLI/SLO tanımlayın: %99.9 uptime, p99 < 500ms gibi
  • Alert fatigue'den kaçının: Sadece actionable alert'ler tanımlayın
  • Dashboard'ları takıma özel oluşturun (overview, detail, debug)

Logging

  • Her zaman structured logging kullanın
  • Hassas veriyi loglarda maskeleyeyin (PII, şifreler)
  • Log retention policy belirleyin (30, 60, 90 gün)
  • Production'da minimum Information seviyesinde loglayın

Tracing

  • Production'da sampling kullanın (%1-%10 arası)
  • İş mantığı span'leri ekleyin (sadece HTTP span'leri yetmez)
  • Error span'lere exception detayı eklemeyi unutmayın
  • Trace context propagation'ı tüm servisler arasında sağlayın

Genel

  • Observability'yi uygulama geliştirme sürecinin parçası yapın (shift-left)
  • Monitoring altyapısını da izleyin (meta-monitoring)
  • Runbook'lar hazırlayın: Alert geldiğinde ne yapılacak?
  • Regular olarak chaos engineering pratikleri yapın
