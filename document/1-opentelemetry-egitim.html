📖 OpenTelemetry Nedir?

OpenTelemetry (OTel), Cloud Native Computing Foundation (CNCF) tarafından geliştirilen, uygulamalardan telemetri verisi (traces, metrics, logs) toplamak için kullanılan açık kaynak bir gözlemlenebilirlik (observability) framework'üdür.

🎯 Temel Amaç: Uygulamaların performansını, davranışını ve sağlığını anlamak için standartlaştırılmış bir telemetri veri toplama yöntemi sunmaktır. Vendor-agnostic (satıcıdan bağımsız) bir çözümdür.

OpenTelemetry Öncesi Sorunlar

  • Her monitoring aracı kendi SDK'sını gerektiriyordu (Jaeger SDK, Zipkin SDK, Datadog SDK vb.)
  • Araç değişikliği = Kod değişikliği
  • Farklı veri formatları ve protokoller
  • Vendor lock-in (satıcıya bağımlılık)

OpenTelemetry ile Çözüm

  • Tek bir SDK ile tüm telemetri verisi toplanır
  • OTLP (OpenTelemetry Protocol) standart protokolü
  • Collector üzerinden istenen backend'e veri yönlendirmesi
  • Kod değişikliği yapmadan backend değiştirme imkanı

🏛️ Observability'nin Üç Sütunu (Three Pillars)

🔗

Traces

Bir isteğin (request) tüm sistem boyunca izlediği yolu gösterir. Her servis arasındaki geçişleri ve süreleri kayıt altına alır.

📊

Metrics

Sayısal ölçümler: CPU kullanımı, bellek tüketimi, istek sayısı, response süresi gibi zaman serisi verileri.

📝

Logs

Uygulamanın ürettiği metin tabanlı olay kayıtları. Hata mesajları, bilgi logları ve debug çıktıları.

🧩 OpenTelemetry Bileşenleri

1. API (Arayüz Tanımları)

Telemetri verisi üretmek için kullanılan arayüzler. Kendi başına veri toplamaz, sadece sözleşme tanımlar.

2. SDK (Uygulama)

API'nin gerçek implementasyonu. Veriyi toplar, işler ve dışa aktarır.

3. Instrumentation (Enstrümantasyon)

Uygulamadaki telemetri veri toplama noktaları:

  • Otomatik (Auto-Instrumentation): ASP.NET Core, HttpClient, SQL Client gibi kütüphaneler için otomatik veri toplama
  • Manuel: Kendi iş mantığınız için özel span ve metrik oluşturma

4. Collector (Toplayıcı)

Telemetri verisini alan, işleyen ve farklı backend'lere ileten aracı bileşen:

Telemetri verisini taşımak için kullanılan standart protokol. gRPC ve HTTP/protobuf üzerinden çalışır.

🔗 Trace Kavramları Detaylı

Trace

Bir isteğin başından sonuna kadar olan tüm yolculuğu temsil eder. Birden fazla span'den oluşur.

Span

Bir trace içindeki tek bir işlem birimini temsil eder. Her span'in:

  • Name: İşlemin adı (örn: "GET /api/products")
  • SpanContext: TraceId, SpanId, TraceFlags
  • Parent SpanId: Üst span referansı
  • Start/End Time: Başlangıç ve bitiş zamanı
  • Attributes: Key-value metadata (http.method, http.status_code vb.)
  • Events: Span süresi içinde gerçekleşen olaylar
  • Status: OK, ERROR, UNSET

⚙️ .NET Core'a OpenTelemetry Kurulumu

Adım 1: NuGet Paketlerini Ekleyin

# Temel paketler
dotnet add package OpenTelemetry
dotnet add package OpenTelemetry.Extensions.Hosting

# Traces için
dotnet add package OpenTelemetry.Instrumentation.AspNetCore
dotnet add package OpenTelemetry.Instrumentation.Http
dotnet add package OpenTelemetry.Exporter.OpenTelemetryProtocol

# Metrics için
dotnet add package OpenTelemetry.Instrumentation.Runtime
dotnet add package OpenTelemetry.Exporter.Prometheus.AspNetCore

# Logs için
dotnet add package OpenTelemetry.Exporter.OpenTelemetryProtocol

Adım 2: Program.cs'de OpenTelemetry Yapılandırması

using OpenTelemetry.Trace;
using OpenTelemetry.Metrics;
using OpenTelemetry.Logs;
using OpenTelemetry.Resources;
using System.Diagnostics;

var builder = WebApplication.CreateBuilder(args);

// Servis adını tanımla
var serviceName = "RestApi";
var serviceVersion = "1.0.0";

// Resource tanımı (tüm telemetri verisine eklenir)
var resourceBuilder = ResourceBuilder.CreateDefault()
    .AddService(serviceName: serviceName, serviceVersion: serviceVersion);

// ===== TRACES =====
builder.Services.AddOpenTelemetry()
    .WithTracing(tracing =>
    {
        tracing
            .SetResourceBuilder(resourceBuilder)
            // Otomatik Instrumentation
            .AddAspNetCoreInstrumentation(options =>
            {
                // Health check endpoint'lerini filtrele
                options.Filter = (httpContext) =>
                    !httpContext.Request.Path.StartsWithSegments("/health");
            })
            .AddHttpClientInstrumentation()
            // OTLP Exporter (Collector'a gönder)
            .AddOtlpExporter(options =>
            {
                options.Endpoint = new Uri("http://localhost:14317");
            });
    })
    // ===== METRICS =====
    .WithMetrics(metrics =>
    {
        metrics
            .SetResourceBuilder(resourceBuilder)
            .AddAspNetCoreInstrumentation()
            .AddHttpClientInstrumentation()
            .AddRuntimeInstrumentation()
            // Prometheus endpoint
            .AddPrometheusExporter();
    });

// ===== LOGS =====
builder.Logging.AddOpenTelemetry(logging =>
{
    logging
        .SetResourceBuilder(resourceBuilder)
        .AddOtlpExporter(options =>
        {
            options.Endpoint = new Uri("http://localhost:14317");
        });
});

var app = builder.Build();

// Prometheus metrics endpoint'i
app.MapPrometheusScrapingEndpoint();

// ... diğer middleware'ler
app.Run();
⚠️ Not: Yukarıdaki kod mevcut Program.cs'ye entegre edilecektir. OTel Collector http://localhost:14317 port'unda dinlemektedir (docker-compose'daki port mapping: 14317:4317).

🛠️ Manuel Instrumentation (Özel Span Oluşturma)

ActivitySource ile Span Oluşturma

.NET'te OpenTelemetry span'leri System.Diagnostics.Activity sınıfı üzerinden oluşturulur:

using System.Diagnostics;

public class ProductController : ControllerBase
{
    // Servis için bir ActivitySource tanımla
    private static readonly ActivitySource ActivitySource =
        new("RestApi.ProductController");

    [HttpGet]
    public async Task<IActionResult> GetProducts()
    {
        // Özel span başlat
        using var activity = ActivitySource.StartActivity("GetProducts");

        // Attribute ekle
        activity?.SetTag("product.query.type", "all");

        try
        {
            // Veritabanı sorgusu için alt span
            using var dbActivity = ActivitySource.StartActivity("DatabaseQuery");
            dbActivity?.SetTag("db.system", "sqlite");
            dbActivity?.SetTag("db.statement", "SELECT * FROM Products");

            var products = await _context.Products.ToListAsync();

            dbActivity?.SetTag("db.result.count", products.Count);
            activity?.SetTag("product.result.count", products.Count);

            return Ok(products);
        }
        catch (Exception ex)
        {
            // Hata durumunda span'e hata bilgisi ekle
            activity?.SetStatus(ActivityStatusCode.Error, ex.Message);
            activity?.RecordException(ex);
            throw;
        }
    }
}
💡 İpucu: using statement ile oluşturulan Activity, scope bittiğinde otomatik olarak sonlanır ve exporter'a gönderilir.

ActivitySource'u OpenTelemetry'ye Kaydetme

builder.Services.AddOpenTelemetry()
    .WithTracing(tracing =>
    {
        tracing
            .AddSource("RestApi.ProductController")  // Manuel source kaydı
            .AddAspNetCoreInstrumentation()
            // ... diğer ayarlar
    });

📡 OTel Collector Konfigürasyonu

Mevcut otel-config.yaml dosyasının açıklamalı hali:

# otel-config.yaml

receivers:
  otlp:
    protocols:
      grpc:          # gRPC protokolü (port 4317)
        endpoint: 0.0.0.0:4317
      http:          # HTTP protokolü (port 4318)
        endpoint: 0.0.0.0:4318

processors:
  batch:             # Veriyi toplu gönderim için grupla
    timeout: 5s
    send_batch_size: 1024

exporters:
  debug:             # Konsola debug çıktısı
    verbosity: detailed

  otlp:              # Jaeger'a OTLP ile gönder
    endpoint: jaeger:4317
    tls:
      insecure: true

service:
  pipelines:
    traces:          # Trace pipeline
      receivers: [otlp]
      processors: [batch]
      exporters: [debug, otlp]

    metrics:         # Metric pipeline (opsiyonel)
      receivers: [otlp]
      processors: [batch]
      exporters: [debug]

    logs:            # Log pipeline (opsiyonel)
      receivers: [otlp]
      processors: [batch]
      exporters: [debug]

🔄 Context Propagation (Bağlam Yayılımı)

Distributed tracing'in çalışması için trace context'inin servisler arasında taşınması gerekir.

W3C Trace Context Standardı

HTTP isteklerinde aşağıdaki header'lar otomatik eklenir:

# traceparent header örneği
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01

# Format: version-traceId-spanId-traceFlags
# 00      → version
# 4bf9... → traceId (32 hex karakter)
# 00f0... → spanId (16 hex karakter)
# 01      → traceFlags (sampled)
✅ Otomatik: ASP.NET Core ve HttpClient instrumentation'ları W3C Trace Context propagation'ını otomatik olarak yapar. Ekstra yapılandırma gerekmez.

🧪 Pratik Alıştırma

  1. OpenTelemetry NuGet paketlerini projeye ekleyin
  2. Program.cs'e temel OpenTelemetry yapılandırmasını ekleyin
  3. Docker servislerini başlatın (docker compose up -d)
  4. .NET uygulamasını çalıştırın (dotnet run)
  5. API'ye birkaç istek gönderin: 
    # Product endpoint'lerine istek gönder
curl http://localhost:5000/api/products
curl http://localhost:5000/api/products/1

# Auth endpoint'ine istek gönder
curl -X POST http://localhost:5000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test123"}'
  6. Jaeger UI'da trace'leri görüntüleyin: http://localhost:16686
  7. Service dropdown'ından "RestApi" seçin ve trace'leri inceleyin
