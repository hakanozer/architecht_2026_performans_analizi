⚖️ Active Monitoring vs Passive Monitoring

Monitoring yaklaşımları temelde iki kategoriye ayrılır. Her ikisinin de avantaj ve dezavantajları vardır ve genellikle birlikte kullanılırlar.

🔵 Active Monitoring (Aktif İzleme)

Tanım: Sisteme dışarıdan düzenli olarak istek gönderip yanıtı kontrol etme.

  • Health check endpoint'lerini periyodik olarak çağırma
  • Sentetik (yapay) istekler gönderme
  • Ping, HTTP probing, TCP check
  • Dışarıdan kullanıcı deneyimini simüle etme
  • Uptime monitoring servisleri

Araçlar: Prometheus Blackbox Exporter, Pingdom, UptimeRobot, Synthetic Monitoring

🟢 Passive Monitoring (Pasif İzleme)

Tanım: Uygulama içinden üretilen telemetri verisini toplama ve analiz etme.

  • Uygulama loglarını toplama
  • Metrik endpoint'lerini scrape etme
  • Trace verisi toplama
  • Gerçek kullanıcı trafiğinden veri toplama
  • Event-driven monitoring

Araçlar: Prometheus, OpenTelemetry, ELK Stack, Jaeger

Karşılaştırma Tablosu

Kriter Active Monitoring Passive Monitoring
Veri Kaynağı Sentetik istekler Gerçek kullanıcı trafiği
Sorun Tespiti Proaktif (sorun oluşmadan önce) Reaktif (sorun oluştuktan sonra)
Kapsam Belirli senaryolar Tüm trafik
Kaynak Kullanımı Ek trafik üretir Minimal ek yük
Trafik Olmadığında Çalışmaya devam eder Veri üretmez
Kullanıcı Deneyimi Tahmin eder Gerçek deneyimi gösterir

❤️ Active Monitoring: Health Checks

.NET Core Health Checks

// NuGet paketi
// dotnet add package Microsoft.Extensions.Diagnostics.HealthChecks
// dotnet add package AspNetCore.HealthChecks.Sqlite

// Program.cs
builder.Services.AddHealthChecks()
    // Temel sağlık kontrolü
    .AddCheck("self", () => HealthCheckResult.Healthy())
    // Veritabanı kontrolü
    .AddSqlite(
        builder.Configuration.GetConnectionString("DefaultConnection")!,
        name: "database",
        tags: new[] { "db", "sqlite" })
    // Özel health check
    .AddCheck<CustomHealthCheck>("custom-check");

var app = builder.Build();

// Health check endpoint'leri
app.MapHealthChecks("/health", new HealthCheckOptions
{
    Predicate = _ => true,
    ResponseWriter = UIResponseWriter.WriteHealthCheckUIResponse
});

// Liveness probe (Kubernetes için)
app.MapHealthChecks("/health/live", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("live")
});

// Readiness probe
app.MapHealthChecks("/health/ready", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("ready")
});

Özel Health Check Oluşturma

public class CustomHealthCheck : IHealthCheck
{
    private readonly ApplicationDbContext _context;

    public CustomHealthCheck(ApplicationDbContext context)
    {
        _context = context;
    }

    public async Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext context,
        CancellationToken cancellationToken = default)
    {
        try
        {
            // Veritabanı bağlantısını kontrol et
            var canConnect = await _context.Database.CanConnectAsync(cancellationToken);

            if (canConnect)
            {
                var productCount = await _context.Products.CountAsync(cancellationToken);
                return HealthCheckResult.Healthy(
                    $"Database is healthy. Product count: {productCount}",
                    new Dictionary<string, object>
                    {
                        { "product_count", productCount },
                        { "timestamp", DateTime.UtcNow }
                    });
            }

            return HealthCheckResult.Unhealthy("Cannot connect to database");
        }
        catch (Exception ex)
        {
            return HealthCheckResult.Unhealthy(
                "Health check failed",
                exception: ex);
        }
    }
}

📊 Metrik Toplama Stratejileri

Pull Model (Çekme)

Monitoring sistemi hedef uygulamadan metrikleri belirli aralıklarla çeker.

Pull Model:
┌──────────────┐     HTTP GET /metrics     ┌──────────────┐
│  Prometheus  │  ─────────────────────▶   │  .NET App    │
│  (Scraper)   │  ◀─────────────────────   │  (/metrics)  │
│              │     Metric Response        │              │
└──────────────┘     (every 5s)            └──────────────┘

Avantajlar:
✅ Hedef uygulamanın monitoring sistemini bilmesine gerek yok
✅ Merkezi kontrol (scrape aralığı, hedef yönetimi)
✅ Health check dahili (scrape başarısız = servis down)

Dezavantajlar:
❌ Firewall/NAT arkasındaki hedeflere erişim zor
❌ Kısa ömürlü (ephemeral) işler için uygun değil

Push Model (İtme)

Uygulama metrikleri/logları aktif olarak monitoring sistemine gönderir.

Push Model:
┌──────────────┐     OTLP / HTTP POST     ┌──────────────┐
│  .NET App    │  ─────────────────────▶   │  OTel        │
│  (Exporter)  │                           │  Collector   │
│              │                           │              │
└──────────────┘                           └──────────────┘

Avantajlar:
✅ Firewall dostu (outbound bağlantı yeterli)
✅ Kısa ömürlü işler ve batch job'lar için uygun
✅ Uygulama ne zaman isterse veri gönderebilir

Dezavantajlar:
❌ Uygulama monitoring sistemini bilmeli
❌ DDoS riski (aşırı veri gönderimi)
❌ Hedefin down olduğunu tespit etmek zor

Hybrid Model (Karma)

Hybrid Model (Eğitimimizdeki Yaklaşım):

┌──────────────┐    Push (OTLP)    ┌──────────────┐    Push    ┌──────────────┐
│  .NET App    │──────────────────▶│  OTel        │──────────▶│   Jaeger     │
│              │                   │  Collector   │           │  (Traces)    │
│              │    Pull (/metrics)│              │           └──────────────┘
│              │◀──────────────────│              │
└──────────────┘   Prometheus      └──────────────┘
       │
       │  Push (HTTP)
       ▼
┌──────────────┐
│Elasticsearch │
│  (Logs)      │
└──────────────┘

📝 Log Seviyeleri ve Kullanım Rehberi

Seviye Kullanım Örnek
Trace En detaylı log. Sadece debugging sırasında. Entering method GetProducts with filter: all
Debug Geliştirme sırasında faydalı detaylar. Cache miss for key: products_all
Information Normal uygulama akışını takip etme. User {UserId} logged in successfully
Warning Beklenmeyen ama tehlikeli olmayan durumlar. Response time exceeded 2 seconds for /api/products
Error İşlem başarısız oldu ama uygulama çalışmaya devam ediyor. Failed to create product: duplicate name
Critical Acil müdahale gerektiren kritik hata. Database connection pool exhausted
⚠️ Best Practice: Production ortamında minimum Information seviyesinde loglayın. Debug ve Trace seviyeleri yalnızca geliştirme/hata ayıklama sırasında aktif edilmelidir.

🔗 Log-Trace Korelasyonu

Bir isteğe ait logları ve trace'leri birbirine bağlamak için TraceId ve SpanId kullanılır.

OpenTelemetry ile Otomatik Korelasyon

// Serilog ile TraceId/SpanId enrichment
Log.Logger = new LoggerConfiguration()
    .Enrich.FromLogContext()
    .Enrich.With<ActivityEnricher>()  // TraceId, SpanId otomatik eklenir
    .CreateLogger();

// Özel ActivityEnricher
public class ActivityEnricher : ILogEventEnricher
{
    public void Enrich(LogEvent logEvent, ILogEventPropertyFactory propertyFactory)
    {
        var activity = Activity.Current;
        if (activity != null)
        {
            logEvent.AddPropertyIfAbsent(
                propertyFactory.CreateProperty("TraceId", activity.TraceId.ToString()));
            logEvent.AddPropertyIfAbsent(
                propertyFactory.CreateProperty("SpanId", activity.SpanId.ToString()));
            logEvent.AddPropertyIfAbsent(
                propertyFactory.CreateProperty("ParentSpanId", activity.ParentSpanId.ToString()));
        }
    }
}

Korelasyon Kullanımı

// Kibana'da TraceId ile arama:
// TraceId: "4bf92f3577b34da6a3ce929d0e0e4736"

// Jaeger'da aynı TraceId ile trace görüntüleme:
// http://localhost:16686/trace/4bf92f3577b34da6a3ce929d0e0e4736

// Böylece:
// 1. Kibana'da bir hata logu buldunuz
// 2. TraceId'yi kopyaladınız
// 3. Jaeger'da bu TraceId ile trace'i açtınız
// 4. Hangi servis/işlemde hata oluştuğunu gördünüz
✅ Temel İlke: Her zaman log, trace ve metrik verilerinizi aynı TraceId ile ilişkilendirin. Bu, sorun analizinde "tek cam" (single pane of glass) deneyimi sunar.

🎯 Sampling (Örnekleme) Stratejileri

Yüksek trafikli uygulamalarda tüm trace ve log verisini toplamak maliyetli olabilir. Sampling ile veri hacmi kontrol edilir.

Strateji Açıklama Kullanım Alanı
AlwaysOn Tüm trace'leri topla Geliştirme, düşük trafikli ortamlar
AlwaysOff Hiçbir trace toplama Tracing devre dışı bırakma
TraceIdRatio Belirli yüzdeyi topla (örn: %10) Production ortamları
ParentBased Üst span'in kararına uy Mikroservis zincirleri 
// Sampling konfigürasyonu
builder.Services.AddOpenTelemetry()
    .WithTracing(tracing =>
    {
        tracing
            // Development: Tümünü topla
            // .SetSampler(new AlwaysOnSampler())

            // Production: %10 örnekleme
            .SetSampler(new ParentBasedSampler(
                new TraceIdRatioBasedSampler(0.1)))  // %10

            // ... diðer ayarlar
    });

🧪 Pratik Alıştırma

  1. Health check endpoint'lerini .NET uygulamasına ekleyin
  2. curl http://localhost:5000/health ile sağlık durumunu kontrol edin
  3. ActivityEnricher ile log-trace korelasyonunu test edin
  4. Kibana'da bir hata logunun TraceId'sini bulun ve Jaeger'da trace'i açın
  5. Farklı sampling stratejilerini deneyin ve veri hacmindeki değişimi gözlemleyin
